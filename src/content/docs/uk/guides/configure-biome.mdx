Цей посібник допоможе вам зрозуміти, як налаштувати Biome.
Він пояснює структуру файлу конфігурації Biome та як Biome вирішує свою конфігурацію.
Якщо ви вже знайомі з конфігурацією, можливо, вам варто ознайомитися з [довідником конфігурації](/reference/configuration/), який детально описує всі доступні опції.

Biome дозволяє налаштовувати свою поведінку за допомогою опцій CLI або файлу конфігурації під назвою `biome.json` або `biome.jsonc`.
Ми рекомендуємо створювати файл конфігурації для кожного проекту.
Це забезпечує, що кожен член команди має однакову конфігурацію в CLI та в будь-якому редакторі, який підтримує інтеграцію Biome.
Багато опцій, доступних у файлі конфігурації, також доступні в CLI.

## Структура файлу конфігурації

Файл конфігурації Biome називається `biome.json` або `biome.jsonc`.
Зазвичай він розміщується в кореневій директорії вашого проекту, поруч з `package.json` вашого проекту.

Оскільки Biome є інструментарієм, його конфігурація організована навколо інструментів, які він надає.
На даний момент Biome надає три інструменти: форматувач, лінтер та сортувальник імпортів (також називається організатором імпортів).
Усі ці інструменти ввімкнені за замовчуванням.
Ви можете вимкнути один або кілька з них, використовуючи поле `<tool>.enabled`:

```json title="biome.json"
{
  "$schema": "https://biomejs.dev/schemas/1.9.4/schema.json",
  "formatter": {
    "enabled": false
  },
  "linter": {
    "enabled": false
  },
  "organizeImports": {
    "enabled": false
  }
}
```

Опції, які застосовуються до більше ніж однієї мови, розміщуються у відповідному полі інструменту.
Опції інструменту, специфічні для мови, розміщуються під полем `<language>.<tool>`.
Це також дозволяє перевизначати загальні опції для певної мови.
Ви також можете ввімкнути або вимкнути інструмент на основі мови.
У наступному прикладі ми налаштовуємо загальні опції `formatter.indentStyle` та `formatter.lineWidth` для всіх мов.
Також ми встановлюємо специфічну для JavaScript опцію `quoteStyle` у `javascript.formatter` та перевизначаємо `formatter.lineWidth`.
Ми вимкнули форматувач для файлів JSON.

```json5 title="biome.jsonc"
{
  "formatter": {
    "indentStyle": "space", // за замовчуванням `tab`
    "lineWidth": 100 // за замовчуванням `80`
  },
  "javascript": {
    "formatter": {
      "quoteStyle": "single", // за замовчуванням `double`
      "lineWidth": 120 // перевизначає `formatter.lineWidth`
    }
  },
  "json": {
    "formatter": {
      "enabled": false
    }
  }
}
```

:::note
Biome відноситься до `javascript` як до всіх варіантів мови JavaScript.
Це включає TypeScript, JSX та TSX.
:::


## Вирішення файлу конфігурації

Biome використовує автоматичне виявлення для пошуку найближчого файлу конфігурації.
Він шукає у робочій директорії та у батьківських директоріях, поки не знайде файл `biome.json` або `biome.jsonc`.
Якщо конфігурація не знайдена, тоді застосовуються налаштування за замовчуванням Biome.
Якщо в одній папці присутні обидва файли `biome.json` та `biome.jsonc`, пріоритет буде надано `biome.json`.

Ось приклад:

<FileTree>
- app
  - backend
    - biome.json
    - package.json
  - frontend
    - biome.json
    - legacy
      - package.json
    - new
      - package.json
</FileTree>


- Команди Biome, що виконуються в `app/backend/package.json`, використовуватимуть файл конфігурації `app/backend/biome.json`;
- Команди Biome, що виконуються в `app/frontend/legacy/package.json` та `app/frontend/new/package.json`,
будуть використовувати файл конфігурації `app/frontend/biome.json`;

:::caution
Biome не підтримує вкладені файли `biome.json`, ні в CLI, ні в LSP. [Слідкуйте та допомагайте підтримці у відповідному питанні](https://github.com/biomejs/biome/issues/2228)
:::

:::note
Команди Biome підтримують використання опції `--config-path` та змінної середовища `BIOME_CONFIG_PATH`.
Це дозволяє вам вказати власний файл конфігурації або директорію, де Biome може шукати файл `biome.json` або `biome.jsonc`.
Коли ви використовуєте `--config-path` або `BIOME_CONFIG_PATH`, стандартний процес вирішення файлу конфігурації **відключається**.

Якщо `--config-path` або `BIOME_CONFIG_PATH` вказує безпосередньо на файл, ви можете використовувати інші назви, ніж `biome.json` або `biome.jsonc`.
Biome буде читати `.json` файл, використовуючи стандартний JSON парсер.
Для файлів з іншими розширеннями Biome буде обробляти їх як `.jsonc` файли, використовуючи більш гнучкий JSON парсер, який дозволяє коментарі та кінцеві коми.
:::


## Спільний файл конфігурації

Поле `extends` дозволяє розділити вашу конфігурацію на кілька файлів.
Таким чином, ви можете ділитися загальними налаштуваннями між різними проектами або папками.

Ось приклад того, як ви можете налаштувати свою конфігурацію для розширення файлу конфігурації `common.json`:

```json title="biome.json"
{
  "extends": ["./common.json"]
}
```

Записи, визначені в `extends`, вирішуються з шляху, де визначено файл `biome.json`.
Вони обробляються в порядку, в якому вони перераховані, причому налаштування в пізніших файлах перевизначають попередні.

Biome може вирішувати файли конфігурації з директорії `node_modules/`.
Таким чином, ви можете експортувати свій файл конфігурації з пакету та імпортувати його в кілька проектів.

Для цього перше, що потрібно зробити, це налаштувати вашу "спільну" конфігурацію Biome певним чином.
Припустимо, що ви хочете поділитися конфігурацією з пакету під назвою `@org/shared-configs`, використовуючи специфікатор `@org/shared-configs/biome`.
Вам потрібно створити запис `exports` у `package.json` цього пакету:

```json title="package.json" ins={5,3}
{
  "name": "@org/shared-configs",
  "type": "module",
  "exports": {
    "./biome": "./biome.json"
  }
}
```

Переконайтеся, що `@org/shared-configs` правильно встановлено у вашому проекті, та оновіть файл `biome.json`, щоб він виглядав як наступний фрагмент:

```json title="biome.json"
{
  "extends": ["@org/shared-configs/biome"]
}
```

Biome спробує **вирішити** вашу бібліотеку `@org/shared-configs/` з вашої робочої директорії.
Робоча директорія це:

- при використанні CLI, директорія, з якої ви виконуєте свої скрипти.
  Зазвичай вона відповідає розташуванню вашого файлу `package.json`;
- при використанні LSP, коренева директорія вашого проекту.

:::caution
Щоб уникнути критичних змін у тому, як працює існуюче вирішення, шляхи, що починаються з крапки `.` або закінчуються на `.json` або `.jsonc`, **не будуть** вирішуватися з `node_modules/`.
:::

Для отримання додаткової інформації про алгоритм вирішення, зверніться до [документації Node.js](https://nodejs.org/api/esm.html#resolution-and-loading-algorithm).


## Ігнорування файлів

Перший спосіб контролювати, які файли та директорії обробляються Biome, це перерахувати їх у CLI.
У наступній команді ми форматуватимемо лише `file1.js` та всі файли в директорії `src`.
Директорії рекурсивно обходяться.

```shell
biome format file1.js src/
```

:::caution
Шаблони глобів, що використовуються в командному рядку, не інтерпретуються Biome.
Вони розширюються вашою оболонкою.
Деякі оболонки не підтримують рекурсивний глоб `**`.
:::

Файл конфігурації Biome може бути використаний для уточнення, які файли обробляються.
Ви можете явно перерахувати файли, які потрібно обробити, використовуючи `include`, та файли, які не потрібно обробляти, використовуючи `ignore`.
`include` та `ignore` приймають шаблони глобів, такі як `src/**/*.js`.
Дивіться [відповідний розділ](#пояснення-синтаксису-глобів) для підтримуваних синтаксисів глобів.
`include` завжди застосовується першим перед застосуванням `ignore`.
Це дозволяє вам включити деякі файли та ігнорувати деякі з включених файлів.

:::note
`include` та `ignore` мають трохи різну семантику.
`include` не заважає Biome обходити папку.
Це означає, що якщо ви хочете запобігти обходу папки Biome, вам потрібно додати папку до `ignore`.
:::

Biome надає глобальні поля `files.include` та `files.ignore`, які застосовуються до всіх інструментів.
Ви також можете включати та ігнорувати файли на рівні інструменту, використовуючи `<tool>.include` та `<tool>.ignore`.
Зверніть увагу, що вони не перевизначають глобальні `files.include` та `files.ignore`.
`files.include` та `files.ignore` застосовуються першими перед `include` та `ignore` інструменту.

Розглянемо наступну конфігурацію:

```json title="biome.json"
{
  "files": {
    "include": ["src/**/*.js", "test/**/*.js"],
    "ignore": ["**/*.min.js"],
  },
  "linter": {
    "ignore": ["test"]
  }
}
```

І виконаємо наступну команду:

```shell
biome format test/
```

Команда відформатує файли, які закінчуються на `.js` і не закінчуються на `.min.js` з директорії `test`.
Файли в `src` не форматуються, оскільки директорія не вказана в CLI.

Якщо ми виконаємо наступну команду, жодні файли не будуть перевірені, оскільки директорія `test` явно ігнорується для лінтера.

```shell
biome lint test/
```

Biome вирішує глобальні шаблони відносно робочої директорії.
Робоча директорія - це директорія, де ви зазвичай виконуєте команду CLI.
Це означає, що вам потрібно звернути **особливу увагу**, коли файл конфігурації розміщений в іншій директорії, ніж та, де ви виконуєте команду.
У випадку редактора (LSP) робоча директорія - це коренева директорія вашого проекту.

Розглянемо проект, що містить дві директорії `backend/` та `frontend/`, і файл конфігурації Biome, який ми представили раніше.
Всередині директорії `frontend/` файл `package.json` визначає скрипт `format`, який запускає форматувач Biome.

<FileTree>
- backend
  - ...
- biome.json
- frontend
  - package.json
  - src
    - ...
  - test
    - ...
</FileTree>

```json title="frontend/package.json"
{
  "name": "frontend-project",
  "scripts": {
    "format": "biome format --write ./"
  }
}
```

Коли ви запускаєте скрипт `format` з `frontend/package.json`,
робоча директорія, визначена цим скриптом, буде `frontend/`.
Глобальні шаблони `src/**/*.js` та `test/**/*.js` матимуть як "базову" директорію `frontend/`.
Таким чином, будуть відформатовані лише файли з `frontend/src/` та `frontend/test/`.

```json title="biome.json"
{
  "files": {
    "include": ["src/**/*.js", "src/**/*.ts"],
    "ignore": ["test"]
  },
  "formatter": {
    "indentStyle": "space"
  }
}
```

:::caution
`ignore` та `include` всередині `overrides` мають **різну** семантику:
- для `ignore`: якщо файл відповідає глобам, **не застосовувати** конфігурацію всередині цього перевизначення, і продовжувати застосовувати наступні перевизначення;
- для `include`: якщо файл відповідає глобам, **застосовувати** конфігурацію всередині цього перевизначення, і продовжувати застосовувати наступні перевизначення;
:::

:::note
За замовчуванням, Biome завжди ігнорує деякі файли, які називаються **захищеними файлами**.
Це означає, що жодні діагностики ніколи не будуть видані Biome для цих файлів.
На даний момент, наступні файли є захищеними:

- `composer.lock`
- `npm-shrinkwrap.json`
- `package-lock.json`
- `yarn.lock`
:::

:::note
Ви також можете [ігнорувати файли, які ігнорує ваша VCS](/guides/integrate-in-vcs#use-the-ignore-file).
:::


## Відомі файли

Ось деякі відомі файли, які ми спеціально обробляємо на основі їхніх імен файлів, а не розширень.
На даний момент, відомі файли є лише JSON-подібними файлами, але ми можемо розширити список, щоб включити інші типи, коли ми підтримуватимемо нові парсери.

Наступні файли парсяться як `JSON` файли з обома опціями `json.parser.allowComments` та `json.parser.allowTrailingCommas`, встановленими в `false`.

- `.all-contributorsrc`
- `.arcconfig`
- `.auto-changelog`
- `.bowerrc`
- `.c8rc`
- `.htmlhintrc`
- `.imgbotconfig`
- `.jslintrc`
- `.nycrc`
- `.tern-config`
- `.tern-project`
- `.vuerc`
- `.watchmanconfig`
- `mcmod.info`

Наступні файли парсяться як `JSON` файли з опцією `json.parser.allowComments`, встановленою в `true`, але `json.parser.allowTrailingCommas`, встановленою в `false`. Це тому, що інструменти, які споживають ці файли, можуть лише видаляти коментарі.

- `.eslintrc.json`

Наступні файли парсяться як `JSON` файли з опціями `json.parser.allowComments` та `json.parser.allowTrailingCommas`, встановленими в `true`. Це тому, що інструменти, які споживають ці файли, розроблені для підтримки таких налаштувань.

## Пояснення синтаксису глобів

Глобальний шаблон визначає набір імен файлів.
Biome підтримує наступні глобальні шаблони:

- `*` відповідає нулю або більше символів. Він не може відповідати роздільнику шляхів `/`.
- `**` рекурсивно відповідає директоріям та файлам.
  Ця послідовність **повинна** утворювати один компонент шляху,
  тому і `**a`, і `b**` є недійсними і призведуть до помилки.
  Послідовність більше ніж двох послідовних символів `*` також є недійсною.
- `[...]` відповідає будь-якому символу всередині дужок.
  Діапазони символів також можуть бути вказані, як упорядковані за Unicode, тому, наприклад,
  `[0-9]` вказує будь-який символ між 0 і 9 включно.
- `[!...]` є запереченням `[...]`, тобто відповідає будь-яким символам, **не** вказаним у дужках.

Деякі приклади:

- `dist/**` відповідає директорії dist та всім файлам у цій директорії.
- `**/test/**` відповідає всім файлам у будь-якій директорії з назвою `test`, незалежно від того, де вони знаходяться. Наприклад, `dist/test`, `src/test`.
- `**/*.js` відповідає всім файлам, що закінчуються на `.js` у всіх директоріях.

Biome використовує бібліотеку глобів, яка трактує всі глобальні шаблони як такі, що мають префікс `**/`.
Це означає, що `src/**/*.js` та `**/src/**/*.js` трактуються як ідентичні.
Вони відповідають як `src/file.js`, так і `test/src/file.js`.

:::caution
Ці шаблони можуть бути використані у файлі конфігурації Biome.
Глобальні шаблони, використані в командному рядку, не інтерпретуються Biome.
Вони розширюються вашою оболонкою.
Деякі оболонки не підтримують рекурсивний глоб `**`
:::
