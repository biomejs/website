Biome може надати деякі інструменти, які допоможуть вам правильно використовувати його у великих проєктах, таких як монорепозиторії або робочі простори, що містять кілька проєктів.

## Використання кількох конфігураційних файлів

Коли ви використовуєте функції Biome - або через CLI, або через LSP - інструмент шукає найближчий конфігураційний файл, використовуючи поточний робочий каталог.

Якщо Biome не знаходить конфігураційний файл там, він **починає підніматися вгору** по каталогах файлової системи, поки не знайде його.

Ви можете скористатися цією функцією, щоб застосувати різні налаштування залежно від проєкту/каталогу.

Припустимо, у нас є проєкт, що містить бекенд-додаток і новий фронтенд-додаток.

<FileTree>
  - app
    - backend
      - biome.json
      - package.json
    - frontend
      - biome.json
      - legacy-app
        - package.json
      - new-app
        - package.json
</FileTree>

Це означає, що коли ви запускаєте скрипт з файлу `app/backend/package.json`, Biome використовуватиме конфігураційний файл `app/backend/biome.json`.

Коли ви запускаєте скрипт з `app/frontend/legacy-app/package.json` або `app/frontend/new-app/package.json`, Biome використовуватиме конфігураційний файл `app/frontend/biome.json`.

## Спільне використання конфігурації

Можна використовувати опцію конфігурації [`extends`](/guides/configure-biome/#share-a-configuration-file), щоб розподілити налаштування по файлах.

Припустимо, у нас є такі вимоги:
- `legacy-app` має форматуватися з використанням пробілів;
- `backend` і `new-app` мають форматуватися з використанням табуляцій;
- всі додатки мають форматуватися з шириною рядка 120;
- `backend` додаток потребує додаткового лінтингу;

Почнемо з створення нового конфігураційного файлу в `app/biome.json`, і помістимо туди спільні налаштування:

```json title="app/biome.json"
{
  "formatter": {
    "enabled": true,
    "lineWidth": 120
  }
}
```

Тепер **перемістимо** `app/frontend/biome.json` до `app/frontend/legacy-app/`, тому що саме там нам потрібно використовувати інше форматування.

```json title="app/frontend/legacy-app/biome.json"
{
  "formatter": {
    "indentStyle": "space"
  }
}
```

Потім ми скажемо Biome успадковувати всі налаштування з основного файлу `app/biome.json`, використовуючи властивість `extends`:

```json title="app/frontend/legacy-app/biome.json" ins={2}
{
   "extends": ["../../biome.json"],
   "formatter": {
     "indentStyle": "space"
   }
}
```

Перейдемо до `app/backend/biome.json`, де нам потрібно увімкнути лінтинг:

```json title="app/backend/biome.json"
{
  "extends": ["../biome.json"],
  "linter": {
    "enabled": true,
    "rules": {
      "recommended": true
    }
  }
}
```

## Монорепозиторії

Монорепозиторії - це особливі репозиторії, де кілька бібліотек зберігаються та підтримуються в одному великому репозиторії. Кожна бібліотека представляє собою самодостатній проєкт, який може містити різні конфігурації.

Biome не дуже добре підтримує монорепозиторії через деякі обмеження у вирішенні вкладених конфігураційних файлів, ви можете [допомогти та стежити за відповідною проблемою](https://github.com/biomejs/biome/issues/2228).

Щоб мати найкращий досвід розробки, незважаючи на поточні обмеження, рекомендується мати `biome.json` у корені монорепозиторію та використовувати конфігурацію [`overrides`](/reference/configuration/#overrides), щоб змінити поведінку Biome у певних пакетах.

У наступному прикладі ми вимикаємо правило `suspicious/noConsoleLog` у пакеті `packages/logger`.

```json title="biome.jsonc"
{
  "linter": {
    "enabled": true,
    "rules": {
      "recommended": true
    }
  },
  "overrides": [{
      "include": ["packages/logger/**"],
      "linter": {
        "rules": {
          "suspicious": {
            "noConsoleLog": "off"
          }
        }
      }
  }]
}
```
