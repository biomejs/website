Виправлення правил лінтингу, макетів форматування тощо можуть завадити вашим скриптам пройти перевірку. Через природу цих змін, **дуже рекомендується** зберігати _точну_ версію у вашому `package.json`, замість використання операторів діапазону.

Ця методологія забезпечить, що ваш скрипт не зламається несподівано.

## Семантичне Версіювання

Biome дотримується [семантичного версіювання](https://semver.org/). Через природу Biome як інструментарію, може бути незрозуміло, які зміни вважаються основними, мінорними або патчами. Ось чому Biome використовує наступний гід по версіюванню:

### Патч Реліз

* Виправлення правила лінтингу, яке викликає помилки лінтингу для валідного коду (помилкові спрацьовування)
* Виправлення некоректних пропозицій коду
* Виправлення форматування синтаксису, що призводить до невалідного коду або змінює семантику програми.
* Покращення документації
* Внутрішні зміни, які не змінюють функціональність Biome:
  * Рефакторинг
  * Покращення продуктивності
  * Збільшення або зміна покриття тестами
* Покращення формулювань діагностик або виправлення їх відображення.
* Перевипуски після невдалого релізу
* Зміна форматування встановленого синтаксису.

### Мінорний Реліз

* Додавання нового правила або підвищення існуючого правила лінтингу до стабільної групи, яка не рекомендується за замовчуванням.
* Додавання підтримки лінтингу та форматування для нещодавно введеної мовної функції, навіть якщо це призводить до більшої кількості помилок лінтингу.
* Видалення рекомендованих правил
* Депрекція існуючих правил
* Додавання нових опціональних конфігураційних опцій, які не змінюють форматування або не повідомляють більше помилок лінтингу.
* Додавання нового рекомендованого правила лінтингу або підвищення існуючого правила лінтингу з групи nursery до рекомендованого правила лінтингу у стабільній групі.
* Видалення не-*nursery* правила або пониження правила до групи *nursery*.

### Основний Реліз
* Зміни в конфігурації, які призводять до іншого форматування або більшої кількості повідомлених помилок лінтингу (додавання/видалення опцій, зміна значення за замовчуванням)
* Зміни в публічному API Biome
* Підвищення нових функцій або інструментів, які потребують певної уваги

## Розширення для Visual Studio Code

Visual Studio Code [не підтримує теги попередніх релізів](https://code.visualstudio.com/api/working-with-extensions/publishing-extension#prerelease-extensions) для розширень. Ось чому Biome використовує наступну схему версіювання для розрізнення стабільних та попередніх версій:
* Стабільні релізи використовують парні номери версій: 10, 12, 14, 16, ...
* Попередні версії використовують непарні номери версій: 11, 13, 15, 17, ...
