Лінтер Biome статично аналізує ваш код, щоб знайти та виправити поширені помилки та допомогти вам писати кращий, сучасний код.
Він [підтримує кілька мов](/internals/language-support) і надає загалом [**<NumberOfRules/> правил**](/linter/rules/).

## Правила

Лінтер організований у правила.
Правило видає діагностику, коли виявляє код, що не відповідає його вимогам.
Наприклад, правило [noDebugger](/linter/rules/no-debugger) повідомляє про використання інструкції `debugger` у JavaScript коді.

Правило видає діагностику з рівнем `info`, `warn` або `error`.
Діагностика з рівнем `error` змушує команду завершитися з ненульовим кодом,
тоді як діагностика з рівнем `info` або `warn` не змушує команду завершитися з помилкою.

Ви можете змусити команду, яка видає діагностику `warn`, завершитися з помилкою, використовуючи опцію `--error-on-warnings`:

```shell
biome lint --error-on-warnings ./src
```

За замовчуванням, лінтер Biome запускає лише [**рекомендовані правила**](/linter/rules#recommended-rules).
Щоб вимкнути _всі правила_, ви можете вимкнути рекомендовані правила у вашому конфігураційному файлі Biome.
Це може бути корисно у випадках, коли ви хочете увімкнути лише кілька правил.
Рекомендовані правила видають діагностику з рівнем `error`.

Правила поділяються на групи.
Наприклад, правило `noDebugger` є частиною [групи `suspicious`](/linter/rules#suspicious).
Правила з цієї групи виявляють код, який, ймовірно, є неправильним або марним.
Опис кожної групи можна знайти на [сторінці правил](/linter/rules/).

На відміну від інших лінтерів, ми не надаємо жодних правил, які перевіряють форматування коду.
Цей вид перевірки покривається нашим [форматером коду](/formatter/).

Багато правил надають **виправлення коду**, яке можна автоматично застосувати.
Biome розрізняє **безпечні** та **небезпечні** виправлення коду.

### Безпечні виправлення

Безпечні виправлення гарантують, що вони не змінять семантику вашого коду.
Їх можна застосовувати без явного перегляду.

Щоб застосувати _безпечні виправлення_, використовуйте `--write`:

<PackageManagerBiomeCommand command="lint --write ./src" />

### Небезпечні виправлення

Небезпечні виправлення можуть змінити семантику вашої програми.
Тому рекомендується вручну переглядати зміни.

Щоб застосувати як _безпечні виправлення_, так і _небезпечні виправлення_, використовуйте `--write --unsafe`:

<PackageManagerBiomeCommand command="lint --write --unsafe ./src" />

### Принципи правил

Ми вважаємо, що правила повинні бути інформативними та пояснювати користувачеві, чому правило спрацьовує, і що користувач повинен зробити, щоб виправити помилку.
Правило повинно дотримуватися цих **принципів**:

1. Пояснити користувачеві помилку. Зазвичай це повідомлення діагностики.
2. Пояснити користувачеві **чому** спрацьовує помилка. Зазвичай це реалізується додатковим вузлом.
3. Сказати користувачеві, що вони повинні зробити. Зазвичай це реалізується за допомогою дії коду.
Якщо дія коду не застосовна, примітка повинна сказати користувачеві, що вони повинні зробити, щоб виправити помилку.

Якщо ви вважаєте, що правило не дотримується цих принципів,
будь ласка, [відкрийте проблему](https://github.com/biomejs/biome/issues/new?assignees=&labels=S-To+triage&projects=&template=01_bug.yml&title=%F0%9F%90%9B+%3CTITLE%3E).

## CLI

Наступна команда запускає лінтер на всіх файлах у каталозі `src`:

<PackageManagerBiomeCommand command="lint ./src" />

Команда приймає список файлів і каталогів.

:::caution
Якщо ви передаєте глоб як параметр, ваша оболонка розширить його.
Результат розширення залежить від вашої оболонки.
Наприклад, деякі оболонки не підтримують рекурсивний глоб `**` або чергування `{}` у наступній команді:

```shell
biome lint ./src/**/*.test.{js,ts}
```

Розширення оболонки має вартість продуктивності та обмеження на кількість файлів, які ви можете передати команді.
:::

Для отримання додаткової інформації про всі доступні опції, перегляньте [довідку CLI](/reference/cli#biome-lint).

### Пропустити правило або групу

З версії **v1.8.0**, команда `biome lint` приймає опцію `--skip`, яка дозволяє вимкнути правило або правила, що належать до групи.

Наприклад, наступна команда пропускає всі правила, що належать до групи `style` та правило `suspicious/noExplicitAny`:

```shell
biome lint --skip=style --skip=suspicious/noExplicitAny
```

### Запустити правило або групу

З версії **v1.8.0**, команда `biome lint` приймає опцію `--only`, яка дозволяє запустити одне правило або правила, що належать до групи.

Наприклад, наступна команда запускає лише правило `style/useNamingConvention`, правило `style/noInferrableTypes` та правила, що належать до `a11y`. Якщо правило вимкнено в конфігурації, то його рівень серйозності встановлюється на `error` для рекомендованого правила або `warn` в іншому випадку.

```shell
biome lint --only=style/useNamingConvention --only=style/noInferrableTypes --only=a11y
```

## Конфігурація

Правило можна налаштувати відповідно до ваших потреб.

### Вимкнути правило

Правило увімкнено, якщо його рівень серйозності `error`, `warn` або `info`. Ви можете вимкнути правило за допомогою `off`.

Наступна конфігурація вимикає рекомендоване правило `noDebugger` та вмикає правила `noShoutyConstants` та `useNamingConvention`.

Рівень серйозності `warn` корисний у випадках, коли проводиться рефакторинг і потрібно, щоб CI пройшов. Повідомлення діагностики жовте. Ви можете використовувати `--error-on-warnings`, щоб завершити з кодом помилки, коли спрацьовує правило, налаштоване з `warn`.

Рівень серйозності `info` не впливатиме на код виходу CLI, навіть коли передано `--error-on-warnings`. Колір повідомлення діагностики синій.

```json title="biome.json"
{
  "linter": {
    "rules": {
      "suspicious": {
        "noDebugger": "off",
        "noConsoleLog": "info"
      },
      "style": {
        "noShoutyConstants": "warn",
        "useNamingConvention": "error"
      }
    }
  }
}
```

### Налаштувати виправлення правила

З версії **v1.8.0**, можна налаштувати сутність виправлення, використовуючи опцію `fix`.
Є три опції:

- `none`: правило не видаватиме виправлення коду;
- `safe`: правило видаватиме [безпечне виправлення](#safe-fixes);
- `unsafe`: правило видаватиме [небезпечне виправлення](#unsafe-fixes);

```json title="biome.json"
{
  "linter": {
    "rules": {
      "correctness": {
        "noUnusedVariables": {
          "level": "error",
          "fix": "none"
        }
      },
      "style": {
        "useConst": {
          "level": "warn",
          "fix": "unsafe"
        },
        "useTemplate": {
          "level": "warn",
          "fix": "safe"
        }
      }
    }
  }
}
```

### Опції правил

Деякі правила мають опції.
Ви можете налаштувати їх, змінивши значення правила.

- `level` вказує на рівень серйозності діагностики;
- `options` змінюються в залежності від правила.

```json title="biome.json"
{
  "linter": {
    "rules": {
      "style": {
        "useNamingConvention": {
          "level": "error",
          "options": {
            "strictCase": false
          }
        }
      }
    }
  }
}
```

## Ігнорувати код

Іноді розробник хоче ігнорувати правило лінтера для конкретного рядка коду.
Ви можете досягти цього, додавши коментар про придушення над рядком, який видає діагностику лінтера.

Коментарі про придушення мають такий формат:

```js
// biome-ignore lint: <пояснення>
// biome-ignore lint/suspicious/noDebugger: <пояснення>
```

Де

- `biome-ignore` є початком коментаря про придушення;
- `lint` придушує лінтер;
- `/suspicious/noDebugger`: **необов'язково**, група та назва правила, яке ви хочете придушити;
- `<пояснення>` пояснення, чому правило вимкнено

Ось приклад:

```ts
// biome-ignore lint: причина
debugger;
// biome-ignore lint/suspicious/noDebugger: причина
debugger;
```

Biome не надає коментарів про ігнорування, які ігнорують весь файл.
Однак ви можете [ігнорувати файл, використовуючи конфігураційний файл Biome](/guides/configure-biome/#ignore-files).
Зверніть увагу, що ви також можете [ігнорувати файли, які ігнорує ваша VCS](/guides/integrate-in-vcs#use-the-ignore-file).

## Міграція з інших лінтерів

Багато правил лінтера Biome натхненні іншими лінтерами.
Якщо ви хочете мігрувати з інших лінтерів, таких як ESLint або `typescript-eslint`,
перегляньте [сторінку джерел правил](/linter/rules-sources).
Якщо ви мігруєте з ESLint,
у нас є спеціальний [посібник з міграції](/guides/migrate-eslint-prettier#migrate-from-eslint).
