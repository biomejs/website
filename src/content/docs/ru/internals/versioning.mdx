Исправления правил линтинга, макетов форматирования и т.д. могут помешать вашим скриптам пройти проверку. Из-за природы этих изменений
**настоятельно рекомендуется** сохранять _точную_ версию в вашем `package.json`, вместо использования операторов диапазона.

Эта методология гарантирует, что ваш скрипт не упадет неожиданно.

## Семантическое версионирование

Biome следует [семантическому версионированию](https://semver.org/). Из-за природы Biome как инструментария может быть неясно, какие изменения считаются мажорными, минорными или патчами. Поэтому Biome использует следующее руководство по версионированию:

### Патч-релиз

* Исправление правила линтинга, которое выдает ошибки линтинга для валидного кода (ложные срабатывания)
* Исправление некорректных предложений кода
* Исправление форматирования синтаксиса, которое приводит к невалидному коду или изменяет семантику программы.
* Улучшения документации
* Внутренние изменения, которые не меняют функциональность Biome:
  * Рефакторинг
  * Улучшения производительности
  * Увеличение или изменение покрытия тестами
* Улучшение формулировок диагностики или исправление отображения диагностики.
* Повторные релизы после неудачного релиза
* Изменение форматирования устоявшегося синтаксиса.

### Минорный релиз

* Добавление нового правила или повышение существующего правила линтинга до стабильной группы, которая не рекомендуется по умолчанию.
* Добавление поддержки линтинга и форматирования для недавно представленной языковой функции, даже если это приводит к большему количеству сообщений об ошибках линтинга.
* Удаление рекомендуемых правил
* Устаревание существующих правил
* Добавление новых необязательных опций конфигурации, которые не меняют форматирование и не сообщают о большем количестве ошибок линтинга.
* Добавление нового рекомендуемого правила линтинга или повышение существующего правила линтинга из группы nursery до рекомендуемого правила линтинга в стабильной группе.
* Удаление правила не из группы *nursery* или понижение правила до группы *nursery*.

### Мажорный релиз
* Изменения в конфигурации, которые приводят к другому форматированию или большему количеству сообщений об ошибках линтинга (добавление/удаление опций, изменение значения по умолчанию)
* Изменения в публичном API Biome
* Продвижение новых функций или инструментов, которые требуют внимания

## Расширение Visual Studio Code

Visual Studio Code [не поддерживает теги пре-релизов](https://code.visualstudio.com/api/working-with-extensions/publishing-extension#prerelease-extensions) для расширений. Поэтому Biome использует следующую схему версионирования, чтобы различать стабильные релизы и превью:
* Стабильные релизы используют четные номера версий: 10, 12, 14, 16, ...
* Превью используют нечетные номера версий: 11, 13, 15, 17, ...

## Почему закрепление версии важно

Закрепление версии Biome в вашем `package.json` важно для обеспечения того, чтобы каждый член команды и среды CI
использовали одну и ту же версию Biome. Даже патч-релизы могут привести к немного отличающимся результатам форматирования или линтинга.
