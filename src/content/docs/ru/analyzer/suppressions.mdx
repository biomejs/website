title: Подавление предупреждений

---

Анализатор Biome является основой для [линтера](/linter) и [ассистента](/assist); фактически, оба инструмента имеют много общего.

Среди прочего, они используют один и тот же движок подавления предупреждений, что означает, что вы можете подавить правило линтера точно так же, как и действие ассистента.

С помощью подавления предупреждений можно отключить правило линтера (или действие) для конкретной строки кода, диапазона строк или всего файла.

Подавление достигается с помощью **комментариев подавления**.

## Синтаксис подавления

Комментарии подавления имеют следующий формат:

```js
// biome-ignore lint: <объяснение>
// biome-ignore assist: <объяснение>
// biome-ignore syntax: <объяснение>
// biome-ignore lint/suspicious: <объяснение>
// biome-ignore lint/suspicious/noDebugger: <объяснение>
// biome-ignore lint/suspicious/noDebugger(foo): <объяснение>
// biome-ignore-all lint: <объяснение>
// biome-ignore-start lint: <объяснение>
// biome-ignore-end lint: <объяснение>
```

Разберем подробнее:

- `biome-ignore`, `biome-ignore-all`, `biome-ignore-start` и `biome-ignore-end` — это начало комментария подавления.
- После пробела следует **категория** комментария подавления. Это может быть `lint`, `assist` или `syntax`.
- За категорией следует необязательная группа, или группа и имя, разделенные косой чертой. Например, `/suspicious` или `/suspicious/noDebugger`. Чем больше вы указываете, тем более конкретным (целенаправленным) будет ваше подавление.
- Некоторые правила даже позволяют указывать значения во время подавления. Их можно указать в скобках, например: `(foo)`. Обратитесь к документации правила, чтобы узнать, поддерживает ли правило подавление со значениями.
- `<объяснение>` — объяснение, почему правило отключено.

Если вы не уверены в точной категории правила/действия, вы можете обратиться к странице их документации и использовать их *категорию диагностики*.

:::note
Далее для объяснения комментариев подавления используется JavaScript, и используется категория `lint`.
:::


### Строчные подавления

Они отключают правило линтера для **следующей строки** кода.

В следующем примере комментарий подавления `biome-ignore lint/suspicious/noDebugger: reason` отключит оператор `debugger;` на строке 2, но `debugger` на строке 3 все равно вызовет диагностику:

```js title="file.js" ins={1}
// biome-ignore lint/suspicious/noDebugger: reason
debugger;
debugger;
```

### Подавления верхнего уровня

Они отключают правило линтера для всего файла. Они должны быть размещены **в начале файла** и должны начинаться с `biome-ignore-all`.

Эти комментарии подавления очень полезны, когда вы хотите подавить какое-то правило линтера *для конкретного файла*, и вы не хотите полагаться на переопределение конфигурации для достижения этого.

В следующем примере комментарий подавления `biome-ignore-all lint/suspicious/noDebugger: reason` отключит правило линтера для всех строк в `generated.js`:

```js title="generated.js" ins={1}
// biome-ignore-all lint/suspicious/noDebugger: reason

debugger
debugger

```

Когда комментарий подавления верхнего уровня находится не в начале файла, он считается *неиспользуемым*, и Biome выдаст диагностику с категорией `suppression/unused`.

### Диапазонные подавления

Они отключают правило линтера для определенного _диапазона_ в файле, начиная со строки с начальным комментарием и до строки с конечным комментарием.

Чтобы отметить начало диапазонного подавления, комментарий подавления должен начинаться с `// biome-ignore-start`. Чтобы отметить его конец, комментарий подавления должен начинаться с `// biome-ignore-end`.

Следующий пример отключит правило `lint/suspicious/noDoubleEquals` для строк 2 и 3, но строка 5 вызовет диагностику:

```js name="file.js" ins={1} ins={4}
// biome-ignore-start lint/suspicious/noDoubleEquals: reason
a == b;
c == d;
// biome-ignore-end lint/suspicious/noDoubleEquals: reason
f == g;
```

:::note
Диапазонные подавления должны иметь соответствующее подавление `biome-ignore-end`.
:::

Диапазонные подавления также могут перекрываться. Рассмотрим следующий пример:

```js name="file.js"
debugger;
// biome-ignore-start lint/suspicious/noDebugger: reason
debugger
// biome-ignore-start lint/suspicious/noDoubleEquals: reason
a == b;
c == d;
// biome-ignore-end lint/suspicious/noDoubleEquals: reason
debugger
f == g;
// biome-ignore-end lint/suspicious/noDebugger: reason
```

В приведенном выше коде:

- Оператор `debugger` на строке 1 вызовет диагностику, потому что нет комментария подавления, который его отключает.
- Комментарий `// biome-ignore-start lint/suspicious/noDebugger: reason` на строке 2 начинает отключать `noDebugger` со строки 3 и далее.
- Комментарий `// biome-ignore-start lint/suspicious/noDoubleEquals: reason` на строке 4 начинает отключать `noDoubleEquals` со строки 5 и далее.
- Комментарий `// biome-ignore-end lint/suspicious/noDoubleEquals: reason` на строке 7 завершает подавление `noDoubleEquals`, которое было начато комментарием подавления на строке 4.
- Оператор `debugger` на строке 8 **не** вызывает диагностику из-за комментария подавления на строке 2.
- Оператор `f == g` вызывает диагностику, потому что правило `noDoubleEquals` больше не подавляется.
- Комментарий `// biome-ignore-end lint/suspicious/noDebugger: reason` на строке 7 завершает подавление `noDebugger`, которое было начато комментарием подавления на строке 2.
